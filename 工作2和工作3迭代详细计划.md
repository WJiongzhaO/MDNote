## 一、在现有系统基础上的升级理解

### 1.1 现有系统的核心思想（基于源码分析）

- **Vault 已经存在**：`knowledge-vault.entity` 与 `vault-file.service` 等，说明当前系统已经有“知识库（Vault）+ 文档”这一层抽象，只是启动/导航上还偏“项目/文件”视角。
- **知识片段是 AST 聚合根**：
  - `KnowledgeFragment` 是一个完整的 AST 聚合根，负责：
    - 标题、标签（`title`, `tags`）。
    - AST 节点（`nodes`），支持文本、图片、Mermaid 图、代码块等。
    - 资源依赖（`assetDependencies`），自动从 AST 中抽取图片、Mermaid 依赖。
    - 引用关系：`sourceDocumentId` + `referencedDocuments[]`，已经记录“片段来自哪个文档、被哪些文档引用、是否 connected”。
  - `toMarkdown()` 会把 AST 转回 Markdown，片段本身是可视化编辑 + 结构化存储的混合体。
- **片段的当前用法**：
  - 用例层 `KnowledgeFragmentUseCases` 已经支持：
    - 创建/更新/删除/搜索片段，处理图片复制到全局 `fragments` 存储。
    - 将片段“插入到文档”时，不是直接展开内容，而是返回 `{{ref:fragmentId}}` 这种引用标记。
  - 前端 `KnowledgeFragmentSidebar`：
    - 是一个“侧边片段库”，支持创建、编辑、拖拽、标签筛选、Markdown 搜索。
    - 通过 `expandFragmentReferencesInPreview`，在预览中递归展开 `{{ref:...}}`，但不改变文档源，只在预览里拼接内容。
    - 片段库是“全局路径存储”，通过 `electronAPI.fragment` 管理，说明你已经把“片段库”视为跨项目共享资产。
- **知识图谱能力的现状**：
  - `knowledge-graph-extractor` 目前专注于：从 Markdown 中抽取 **标题、双链 [[...]]、标签 #tag**，生成 `KnowledgeGraph`（节点+边）并转为 Mermaid。
  - 现在的图谱更多关注“文档结构/链接/标签”，还没有把“知识片段”本身纳入图谱。

### 1.2 本轮升级在现有思想上的延展

- **从“可引用内容块”到“知识资产”**：
  - 现有片段已经有来源、引用文档列表，这是天然的“关系图”和“使用统计”基础。
  - 升级的重点是：给片段补充元数据（可信度、状态、分类）、生命周期（active/archived/deprecated）、健康度和推荐逻辑。
- **从“文档图谱”到“资产图谱 + 健康度”**：
  - 在现有 `KnowledgeGraph` 之上，把片段/模板变量/父子片段也视为图中的一类“节点”，结合 `referencedDocuments` 和 `derivedFromId` 等信息，形成更完整的资产图谱。
- **与现有 UI/用例的关系**：
  - 不推翻当前 `KnowledgeFragmentSidebar`、`useKnowledgeFragments`、`KnowledgeFragmentUseCases` 等，而是在其上：
    - 增加新的字段和查询能力。
    - 增加一个“片段管理页”（资产视角），保留当前侧边栏作为“快速工具”。
    - 复用已经存在的 `referencedDocuments` 和 `{{ref:...}}` 机制，落地健康度和影响分析。
